Validator functions in BEL SOIL check schema compliance for every SOIL output type. Run them before passing objects to downstream systems, export pipelines, or any integration that expects a governed SOIL contract. Each validator inspects the object's required governance fields, prohibited values, and schema-version markers — and returns a structured result rather than throwing.

## Import Path

```typescript
import {
  validateSoilCompletionSectorPackConfig,
  validateSoilCompletionScenarioRunRequest,
  validateExportReadyObjectPackage,
  // ...etc
} from 'functions/src/soil/services';
```

---

## Validation Return Pattern

Every validator returns a `{ valid: boolean; errors: string[] }` object. When `valid` is `true`, `errors` is an empty array. When `valid` is `false`, `errors` contains one or more human-readable descriptions of what failed.

```typescript
const result = validateSoilCompletionSectorPackConfig(config);

if (!result.valid) {
  console.error('Validation failed:', result.errors);
  // Route to safe failure path — do not pass config downstream
} else {
  // Proceed with the governed pipeline
}
```

<Note>
  Validators never throw. A structurally broken object produces `valid: false` with descriptive errors — not an uncaught exception. Always check `result.valid` before continuing a pipeline run.
</Note>

## Core Validators

### `validateSoilCompletionSectorPackConfig(config)`

Validates a `SoilCompletionSectorPackConfig` against the v2 schema contract.

**Input** `SoilCompletionSectorPackConfig`

**Checks**
- `decisionSupportOnly` is present and `true`
- `humanReviewRequired` is present and `true`
- `sectorPackVersion` is `v2`
- No prohibited output fields (legal determinations, certification, compliance-pass language) are present
- `approvalStatus` is within the allowed posture vocabulary

```typescript
import { validateSoilCompletionSectorPackConfig } from 'functions/src/soil/services';

const result = validateSoilCompletionSectorPackConfig(config);
if (!result.valid) {
  console.error('SectorPackConfig invalid:', result.errors);
}
```

---

### `validateSoilCompletionScenarioRunRequest(request)`

Validates a `SoilCompletionScenarioRunRequest` before it is submitted to the scenario run builder.

**Input** `SoilCompletionScenarioRunRequest`

**Checks**
- Required identifiers are present: `scenarioId`, `soilSessionId`, `tenantId`, `organizationId`, `programId`
- `sectorPackId` is a known, active sector family (not `requires_review`)
- `sectorPackVersion` is `v2`
- `decisionSupportOnly` and `humanReviewRequired` are both `true`
- `conditions` array is non-empty and each condition carries a valid `conditionId`, `conditionVersion`, `confidence` (0–1), and non-empty `assumptions`
- No identity-rich fields are present in condition inputs

```typescript
import { validateSoilCompletionScenarioRunRequest } from 'functions/src/soil/services';

const result = validateSoilCompletionScenarioRunRequest(request);
if (!result.valid) {
  const failure = buildSoilCompletionSafeFailureResponse(
    result.errors,
    request.auditReference,
    request.correlationId
  );
  return failure;
}
```

<Warning>
  If `validateSoilCompletionScenarioRunRequest` returns `valid: false`, route immediately to `buildSoilCompletionSafeFailureResponse`. Do not attempt to construct a scenario run from an invalid request.
</Warning>

## Environmental and Map Validators

### `validateEosEnvironmentalConditionInput(input)`

Validates a single `SoilCompletionEnvironmentalConditionInput` before it is passed to `buildEosEnvironmentalOutput`.

**Input** `SoilCompletionEnvironmentalConditionInput`

**Checks**
- Required fields are present: `conditionId`, `conditionVersion`, `category`, `intensity`
- `confidence` is a number between 0 and 1
- No GIS-rendering or geocoding fields are present — EOS inputs are contract-only

```typescript
import { validateEosEnvironmentalConditionInput } from 'functions/src/soil/services';

for (const input of conditionInputs) {
  const result = validateEosEnvironmentalConditionInput(input);
  if (!result.valid) {
    console.error(`Condition ${input.conditionId} invalid:`, result.errors);
  }
}
```

### `validateMapConsumableOutput(output)`

Validates a `SoilCompletionMapConsumableOutput` data contract.

**Input** `SoilCompletionMapConsumableOutput`

**Checks**
- `mapRenderingIncluded` is `false` — the object must not contain GIS rendering artifacts
- Schema version matches `map_consumable_output_v1`
- Required location signal references are present
- No geocoding, tile, or frontend layer fields are present

```typescript
import { validateMapConsumableOutput } from 'functions/src/soil/services';

const result = validateMapConsumableOutput(mapOutput);
if (!result.valid) {
  console.error('Map output invalid:', result.errors);
}
```

### `validateExecutiveReadinessAssessment(assessment)`

Validates a `SoilCompletionExecutiveReadinessAssessment` and its three ledgers.

**Input** `SoilCompletionExecutiveReadinessAssessment`

**Checks**
- `assessmentId`, `scenarioRunId`, `correlationId`, and `auditReference` are present
- All three ledgers are present: `evidenceLedger`, `assumptionLedger`, `confidenceLedger`
- `approvalStatus` is within the allowed posture vocabulary — prohibited values include `certified`, `authorized`, `compliance_pass`
- Schema version matches `executive_readiness_assessment_v1`
- `decisionSupportOnly` and `humanReviewRequired` are `true`

```typescript
import { validateExecutiveReadinessAssessment } from 'functions/src/soil/services';

const result = validateExecutiveReadinessAssessment(executiveAssessment);
if (!result.valid) {
  console.error('Executive assessment invalid:', result.errors);
}
```

<Warning>
  The executive assessment validator explicitly rejects any object where `approvalStatus` contains certification, authorization, or external-validation-pass language. Only the governed posture vocabulary is accepted. See the [Type Contracts](/sdk/type-contracts) reference for the full list of allowed values.
</Warning>

### `validateExportReadyObjectPackage(pkg)`

Validates the complete `SoilCompletionExportReadyObjectPackage` — the final governed envelope for a pipeline run.

**Input** `SoilCompletionExportReadyObjectPackage`

**Checks**
- `packageId`, `correlationId`, and `auditReference` are present
- All required top-level objects are present: `scenarioRun`, `adminHubOutput`, `executiveAssessment`, `observabilityTrace`, `protectedUseStatus`
- `approvalStatus` is within the allowed posture vocabulary
- `decisionSupportOnly` and `humanReviewRequired` are `true` on the package-level fields
- No prohibited unsafe output fields are present anywhere in the package

```typescript
import { validateExportReadyObjectPackage } from 'functions/src/soil/services';

const result = validateExportReadyObjectPackage(exportPackage);
if (!result.valid) {
  console.error('Export package invalid:', result.errors);
} else {
  console.log('Package is valid and ready for export.');
}
```

<Tip>
  Run `validateExportReadyObjectPackage` as the final gate before handing off a pipeline run to any external consumer. It validates the package-level fields and checks that each required child object is structurally present — saving you from running each child validator individually in the final gate check.
</Tip>
